# @tiles-bot/casper-x402-client

Minimal TypeScript client for building Casper x402 payment headers. It parses Casper `PaymentRequirements`, builds the Casper-native `TransferAuthorization` typed data, signs it with an ed25519 or secp256k1 Casper key, and returns the base64 `X-PAYMENT` header payload.

## Usage

```ts
import {
  createCasperPaymentHeader,
  createCasperSigner,
} from '@tiles-bot/casper-x402-client';

const signer = createCasperSigner(process.env.CASPER_PRIVATE_KEY!, 'ed25519');
const header = await createCasperPaymentHeader(signer, paymentRequirements);

await fetch(resourceUrl, {
  headers: { 'X-PAYMENT': header },
});
```

`createCasperPaymentHeader()` builds the EIP-712 domain from `paymentRequirements.asset`, which must be the wCSPR contract package hash (`hash-...`, `0x...`, or raw 32-byte hex).

## Domain safety

Do not use a zero-address `verifyingContract`. Casper-native wCSPR authorizations should bind signatures to the actual CEP-18 package hash:

```ts
import { createCasperEIP712Domain } from '@tiles-bot/casper-x402-client';

const domain = createCasperEIP712Domain(
  'casper:casper',
  'hash-<64 hex chars for the wCSPR contract package>',
);
```

If you pass an EVM-style domain with `verifyingContract: 0x0000000000000000000000000000000000000000`, signing throws before producing a bad-but-valid-looking signature. That placeholder was a foot-gun with a bayonet; it is intentionally gone.

## Signature format

Casper secp256k1 signatures are emitted in compact Casper format: `r(32) + s(32)` (64 bytes), with no Ethereum recovery byte. ed25519 signatures are 64 bytes.

## Build artifacts

`dist/` is generated by `npm run build` and intentionally ignored by git. The package has a `prepack` script so npm packaging builds fresh artifacts instead of relying on checked-in generated files.

This package is intentionally standalone rather than a root npm workspace package today, so it keeps its own `package-lock.json` for reproducible `npm ci` inside `packages/casper-x402-client/`.
